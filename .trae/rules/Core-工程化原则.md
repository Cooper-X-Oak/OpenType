# Core-工程化原则 (Engineering Principles)

## 1. 记录与验证 (Documentation & Verification)

- **先记录，后执行**：遇到问题或启动任务时，必须先在文档（如 OpenSpec Tasks, Issue）中记录，再进行测试或修复。
- **验证驱动**：所有关键功能必须通过 E2E 测试或脚本进行验证。不可验证的结论是无效的。
- **TDD (测试驱动)**：修复 Bug 前，必须先编写能复现问题的失败测试用例。
- **根因分析 (5 Whys)**：不要停留在表面原因，应进行至少 5 次追问，挖掘系统性根因。

## 2. 终端与网络健壮性 (Terminal & Network)

- **闭环原则**：所有长运行进程必须明确管理生命周期（提供停止方式），脚本必须有明确的退出码（成功0/失败非0）。
- **网络健壮性**：
  - **镜像源**：涉及二进制下载（Puppeteer, Electron 等）必须配置 `.npmrc` 使用国内镜像（阿里云/腾讯云），严禁直接依赖可能受阻的 CDN。
  - **超时控制**：网络请求必须设置超时（建议 10-30s），超时后抛出清晰错误，严禁无限挂起。
- **环境检测**：重型操作前（如启动浏览器）先检测环境（端口占用等）。

## 3. 脚本与工具管理 (Scripts & Tools)

- **脚本持久化**：有价值的测试/调试脚本应持久化保存（如在 `scripts/` 或 `.trae/script_tools/`），禁止用后即焚。
- **工具使用**：配置复杂工具（如 MCP, Linter）时，优先参考官方文档和最佳实践，确保配置文件的路径和格式正确。

## 4. Git 工作流与网络适配 (Git & Network)

- **原子性自动提交 (Atomic Auto-Commit)**：
  - **原则**：Agent 必须在完成一个完整的变更单元（Feature/Fix/Refactor）并通过验证后，**主动**执行提交，无需等待用户指令。
  - **时机**：OpenSpec Task 完成时、Bug 修复验证通过时、重构步骤完成时。
  - **禁止**：代码处于中间状态（编译失败、逻辑不完整）时提交。
- **智能提交工具**：必须使用 `git-smart-commit` skill 生成符合 Conventional Commits 的中文提交信息。
- **网络感知 Push**：在中国大陆网络环境下，Push 代码必须具备网络感知能力。
  - **代理检测**：Push 失败时应自动尝试本地代理（如 Clash 7890 端口）。
  - **配置隔离**：推荐使用 `git -c` 参数临时指定代理，避免污染全局 git config。
